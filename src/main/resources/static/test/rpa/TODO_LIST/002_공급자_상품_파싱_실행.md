# 002 공급자 상품 파싱 실행

## 목적
- 준비된 `TrendKeyword` 목록을 기준으로 공급자와 상품 저장을 순차 실행한다.

## 1. 공급자 저장 호출
- 최근 저장된 `TrendKeyword` 20건을 기준으로 공급자 저장 API를 반복 호출한다.
- 예시 URL은 `http://localhost:8080/api/Products/saveWithSequentialId?keywordId=160&rankId=2179193963&syncDate=20260428`다.
- 현재 공급자 조회 API는 `/api/Suppliers?rankId=...&syncDate=...`로 구현되어 있다.
- 현재 공급자 저장 API는 `/api/Suppliers/saveWithSequentialId?rankId=...&syncDate=...`로 구현되어 있다.

## 2. 상품 저장 호출
- 공급자 저장과 같은 흐름으로 상품 저장까지 연결한다.
- 현재 상품 조회 API는 `/api/Products?rankId=...&syncDate=...`로 구현되어 있다.
- 현재 상품 저장 API는 `/api/Products/saveWithSequentialId?keywordId=...&rankId=...&syncDate=...`로 구현되어 있다.
- 현재 상품 저장 메서드는 내부에서 공급자를 먼저 저장한 뒤 상품을 저장한다.
- 현재 `V3__create_supplier_and_product_table.sql` 기준으로 `product.sync_date DATETIME` 컬럼이 존재한다.

## 3. 반복 처리 규칙
- 반복 시 조합할 값은 `keywordId`, `rankId`, `syncDate`다.
- `TrendKeyword -> Supplier -> Product` 순서로 처리한다.
- 하드코딩된 값은 예시로만 사용하고, 실제 구현에서는 동적으로 조회한다.
- 실제 구현에서는 `TrendKeyword.rankingId`를 `rankId`로, `TrendKeyword.syncDate`를 `syncDate`로 변환해 사용해야 한다.
- `SupplierAndProductService`는 `Constants.Snxbest.API_V1_FOOD_PRODUCT_DATA_URL_TEMPLATE`를 사용해 네이버 공급자/상품 API를 호출한다.
- `Product.syncDate`에는 해당 상품이 네이버 API의 어떤 동기화 날짜 기준으로 저장되었는지 기록해야 한다.

## 4. 중단 재개 기준
- 중단 재개를 고려해 각 단계의 진행 상황을 기록한다.
- 당일 파싱한 `TrendKeyword` 목록을 기준으로 반복문을 돌린다.

## 완료 기준
- 당일 `TrendKeyword` 목록을 기준으로 공급자와 상품 저장이 반복 실행된다.
- 샘플 파라미터가 아닌 실제 조회값으로 저장 API가 호출된다.

## 추가 구현 필요
- 당일 `TrendKeyword` 목록 20건을 순회하면서 `/api/Products/saveWithSequentialId`에 해당하는 서비스 로직을 직접 호출하는 RPA orchestration 서비스가 필요하다.
- 현재 개별 테스트용 API는 존재하지만, 전체 20건을 자동으로 이어 실행하는 단일 RPA 실행 로직은 아직 없다.
- `SupplierAndProductService.saveSuppliersWithSequentialId()`는 공급자만 저장하고 상품 저장은 하지 않으므로, 최종 시퀀스는 `saveSupplierAndProductsWithSequentialId()` 중심으로 정리하는 편이 안전하다.
