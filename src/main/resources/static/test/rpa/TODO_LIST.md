# semi_food RPA TODO 리스트

이 문서는 `semi_food` 프로젝트의 현재 `RPA` 도메인 구조에 맞춰 정리한 작업 목록입니다.

## 1단계: 데이터베이스 및 모델 설계
- `rpa_log.status` 기준 상태 컬럼 설계: `RUNNING`, `COMPLETED`, `FAILED`
- 작업 시작 시각(`started_at`) 및 종료 시각(`ended_at`) 기록 컬럼 추가
- `RpaLog`를 중심으로 자동화 수행 중 발생한 상세 메시지와 요약/에러를 저장
- 상태 변경 이력과 재처리 가능 여부를 함께 고려한 테이블 구조 정리

## 2단계: 백엔드 로직(Spring Boot) 구현
- 낙관적 락(Optimistic Lock) 적용: `@Version` 기반 동시성 제어로 중복 실행 방지
- 비동기 처리: `@Async`를 활용해 웹 요청과 실제 RPA 처리 로직을 분리
- CRUD 방어: `RUNNING` 상태일 때 수정/삭제 요청이 들어오면 예외를 발생시키는 가드 로직 작성
- 예외 처리 보장: `try-finally`로 성공/실패와 무관하게 상태 해제 및 종료 처리를 보장

## 3단계: 기아 상태(Starvation) 방지 장치
- 자동 복구 스케줄러: `@Scheduled`로 장시간 `RUNNING` 상태인 데이터를 찾아 강제 초기화
- 재시도 전략: 외부 사이트의 일시적 오류를 대비해 3회 내외의 자동 재시도 로직 추가
- 장시간 점유 데이터에 대한 감시 기준과 복구 기준 시간 정의

## 4단계: 사용자 UI/UX 및 대시보드
- 데이터 목록 UI에서 처리 중 데이터에 `작업 중` 표시를 노출하고 수정/삭제 버튼 비활성화
- 관리자 대시보드 구축: 진행/완료/실패 상태별 통계 노출
- 강제 잠금 해제(Unlock) 버튼 구현으로 기아 상태 수동 복구 지원
- 상세 에러 로그 조회 팝업을 통해 `RpaLog` 기반 오류 내용을 확인 가능하게 구성

## 5단계: 선택 항목 및 고도화
- 실패 알림 검토: 작업 실패 또는 장시간 점유 시 관리자에게 메일/슬랙 알림 발송
- 운영 편의성을 위한 알림 이력과 재처리 이력 관리 방안 검토
- 향후 확장 가능한 RPA 작업 유형별 공통 처리 구조 정리
