# 006 비동기 동시성 제어

## 목적
- 웹 요청과 실제 RPA 실행을 분리하고, 중복 실행으로 인한 충돌을 방지한다.

## 1. 비동기 처리
- `@Async`를 사용해 웹 요청과 실제 RPA 실행을 분리한다.
- 현재 프로젝트에서 `@Async` 또는 `@EnableAsync` 구현은 확인되지 않았다.
- 현재 `/rpa/run`은 placeholder 응답만 반환한다.

## 2. 동시성 제어
- 중복 실행 방지를 위해 낙관적 락(`@Version`)을 검토한다.
- 현재 `RpaLog`, `TrendKeyword`, `Supplier`, `Product` 엔티티에는 `@Version` 필드가 없다.
- 현재 중복 실행 방지는 구현되어 있지 않다.

## 3. 종료 보장
- `try-finally` 구조를 사용해 성공/실패와 관계없이 종료 처리를 보장한다.
- 현재 `RpaLog.complete()`와 `RpaLog.fail()`은 있으나, RPA 전체 실행 흐름에서 호출하는 로직은 아직 없다.

## 4. 재시도
- 외부 사이트 호출은 실패 가능성이 있으므로 재시도 전략을 둔다.

## 완료 기준
- RPA 실행 요청이 웹 요청과 분리되어 처리된다.
- 중복 실행과 예외 발생 시 상태 꼬임을 방지한다.

## 추가 구현 필요
- `@EnableAsync` 설정 클래스를 추가하거나 메인 애플리케이션에 활성화해야 한다.
- 최신 `RpaLog.status == RUNNING`이면 새 실행을 거부하는 1차 가드가 필요하다.
- `@Version` 도입 전에는 수동 ID 정책, TiDB 동작, 기존 마이그레이션과의 충돌을 먼저 검토해야 한다.
- RPA 실행 서비스는 시작 로그 저장, 파싱 실행, 완료/실패 상태 갱신을 하나의 흐름으로 가져야 한다.
