# 008 완료 기준 및 주의사항

## 전체 완료 기준
1. `rpa_log` 기준 상태와 로그 구조가 확정되어야 한다.
2. 파싱 RPA 시퀀스가 먼저 동작해야 한다.
3. CRUD 안전 장치가 적용되어야 한다.
4. 관리자 대시보드가 구현되어야 한다.
5. SSE 실시간 반영과 알림 기능은 후순위로 확장한다.

## 현재 구현 상태 요약
- 개별 파싱 테스트 API는 `ParserController`에 구현되어 있다.
- `TrendKeywordService`는 네이버 트렌드 키워드 조회와 저장을 수행한다.
- `/api/TrendKeywords/rpaTargets?size=20`로 당일 기준 후속 파싱 입력값을 조회할 수 있다.
- `SupplierAndProductService`는 네이버 공급자/상품 조회, 공급자 저장, 상품 저장을 수행한다.
- `/api/Products/rpaSaveToday?size=20`로 당일 `TrendKeyword` 타깃 기준 공급자/상품 저장을 반복 실행할 수 있다.
- `RpaLog`는 공급자/상품 RPA 실행 흐름에 연결되어 시작/완료/실패 요약을 저장한다.
- `RpaParsingLogFileService`는 실행 상세 결과를 `src/main/resources/static/test/rpa/log`에 파일로 저장한다.
- `RpaRestController`, `static/rpa/dashboard.html`, SSE, Async, Scheduler, CRUD 가드는 아직 미완성이다.

## 구현 시 주의사항
- 예시 URL의 `localhost`, `keywordId`, `rankId`, `syncDate` 값은 샘플일 뿐이며, 실제 구현은 동적 값 기준으로 작성한다.
- 문서에 나온 모든 항목을 한 번에 완성하려고 하지 말고, 실제 구현 순서대로 나눠서 진행한다.
- 이 문서는 설계 방향을 정리한 것이므로, 실제 코드 작성 시에는 현재 엔티티와 마이그레이션 스키마를 우선한다.
- `TrendKeyword.syncDate`와 `Product.syncDate`는 현재 마이그레이션 기준으로 `DATETIME` 컬럼과 매핑된다.
- 공급자 테이블에는 `sync_date`가 없으므로 공급자 조회/삭제 기준은 `createdAt` 또는 상품과의 연관 관계 기준으로 별도 결정해야 한다.
- 수동 ID 부여 정책이 여러 엔티티에 적용되어 있으므로 자동 증가 전략을 전제로 코딩하면 안 된다.
- 현재 SecurityConfig는 개발 편의를 위해 대부분 요청을 허용하고 있으므로 관리자 전용 기능 구현 시 접근 제어를 반드시 함께 검토해야 한다.
