# 003 로그 저장

## 목적
- RPA 실행 상태와 상세 디버깅 정보를 DB 로그와 파일 로그로 분리해 남긴다.

## 1. DB 로그 저장
- DB에는 `message`를 통해 어떤 데이터가 저장되었는지 요약 기록한다.
- `rpa_log`의 `status`, `started_at`, `ended_at`, `keyword_count`, `product_count`를 함께 사용한다.
- 현재 `RpaLog` 엔티티와 `RpaLogRepository`는 존재한다.
- 현재 `RpaLog.builder(startedAt)`는 생성 시 상태를 `RUNNING`으로 설정한다.
- 현재 `RpaLog.complete(keywordCount, productCount)`와 `RpaLog.fail(message)`가 구현되어 있다.
- `RpaSupplierProductParsingService.parseTodaySupplierAndProducts()` 실행 시 `RpaLog`가 `RUNNING`으로 먼저 저장된다.
- 실행 종료 후 실패가 없으면 `COMPLETED`, 실패가 하나라도 있으면 `FAILED`로 갱신된다.
- `RpaLogRepository.findMaxId()`가 추가되어 TiDB 수동 ID 정책에 맞게 로그 ID를 부여한다.

## 2. 파일 로그 저장
- 파일 로그에는 RPA 실행 시점, 저장된 데이터, 디버깅 로그를 상세히 기록한다.
- 로그 파일 저장 위치는 `src/main/resources/static/test/rpa/log`다.
- 로그 파일명 규칙은 `rpa_parsing_yymmdd_time.log`다.
- `RpaParsingLogFileService`가 추가되어 실행마다 파일 로그를 저장한다.
- 실제 파일명 형식은 `rpa_parsing_yyMMdd_HHmmss.log`다.

## 3. 로그 역할 분리
- DB 로그는 운영 조회와 상태 추적용으로 사용한다.
- 파일 로그는 디버깅과 상세 추적용으로 사용한다.

## 완료 기준
- RPA 실행마다 `rpa_log`에 요약 상태가 남는다.
- 실행마다 지정된 위치에 상세 로그 파일이 남는다.
- DB 로그의 `message`에 파일 로그 경로가 함께 기록된다.

## 추가 구현 필요
- 현재 `rpa_log` 날짜 범위 조회와 대시보드용 목록 조회 메서드는 아직 필요하다.
- 파일 로그 저장 실패 시 DB 로그에는 `FILE_LOG_WRITE_FAILED` 메시지가 남지만, 별도 장애 알림은 아직 없다.
- 현재 로그는 공급자/상품 파싱 실행 흐름에 연결되어 있으며, 트렌드 키워드 저장 단계까지 포함한 전체 RPA 로그는 후속 통합 단계에서 확장해야 한다.

## 이번 단계 구현 결과
- `RpaParsingLogFileService`를 추가했다.
- `RpaLogRepository.findMaxId()`를 추가했다.
- `RpaSupplierProductParsingService`에 RPA 시작/종료 DB 로그 저장을 연결했다.
- 키워드별 결과를 파일 로그에 기록하고, DB `message`에 파일 로그 경로를 남긴다.
