# 007 복구 고도화

## 목적
- 장시간 실행 중 상태와 외부 사이트 일시 오류에 대응할 수 있는 복구 체계를 준비한다.

## 1. 자동 복구
- `@Scheduled`를 사용해 장시간 `RUNNING` 상태인 데이터를 찾아 복구한다.
- 현재 프로젝트에서 `@Scheduled` 구현은 확인되지 않았다.
- 현재 장시간 `RUNNING` 상태를 판별하는 Repository 메서드도 없다.

## 2. 재시도 강화
- 외부 사이트의 일시적 오류에 대비해 3회 내외 재시도 로직을 둔다.
- 현재 `RestClient` 호출은 `TrendKeywordService`, `SupplierAndProductService`에 직접 구현되어 있으며 재시도 로직은 없다.

## 3. 알림 확장
- 실패 알림은 메일 또는 슬랙 같은 관리자 채널로 확장 가능하게 설계한다.
- 현재 메일/슬랙 의존성이나 알림 서비스는 확인되지 않았다.

## 4. 이력 관리
- 알림 이력과 재처리 이력을 남길 수 있는 구조를 검토한다.

## 완료 기준
- 장시간 `RUNNING` 상태를 자동으로 탐지하고 복구할 수 있다.
- 실패와 재처리 흐름을 추적할 수 있는 확장 지점이 있다.

## 추가 구현 필요
- 복구 기준 시간을 먼저 정해야 한다. 예: `startedAt`이 현재 시각보다 1시간 이상 오래된 `RUNNING` 로그.
- 복구 시 `FAILED`로 종료할지, 별도 상태 없이 `message`에 강제 종료 사유를 남길지 결정해야 한다.
- 재시도는 라이브러리 도입 없이 간단한 반복문으로 시작하거나, 새 라이브러리 도입 시 별도 승인과 충돌 검토가 필요하다.
