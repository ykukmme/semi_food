# 002 공급자 상품 파싱 실행

## 목적
- 준비된 `TrendKeyword` 목록을 기준으로 공급자와 상품 저장을 순차 실행한다.

## 1. 공급자 저장 호출
- 최근 저장된 `TrendKeyword` 20건은 제품 20개를 의미하지 않고, 서로 다른 20개 키워드 분류를 의미한다.
- 각 키워드 분류마다 공급자/상품 API를 1회 호출하고, 한 번의 API 응답에 들어있는 공급자와 상품을 저장한다.
- 예시 URL은 `http://localhost:8080/api/Products/saveWithSequentialId?keywordId=160&rankId=2179193963&syncDate=20260428`다.
- 현재 공급자 조회 API는 `/api/Suppliers?rankId=...&syncDate=...`로 구현되어 있다.
- 현재 공급자 저장 API는 `/api/Suppliers/saveWithSequentialId?rankId=...&syncDate=...`로 구현되어 있다.
- `RpaSupplierProductParsingService`가 추가되어 당일 RPA 타깃을 기준으로 공급자/상품 저장을 반복 실행한다.
- 공급자는 날짜 기준으로 새로 만들지 않고, 정규화된 공급자 이름이 이미 있으면 기존 공급자를 재사용한다.

## 2. 상품 저장 호출
- 공급자 저장과 같은 흐름으로 상품 저장까지 연결한다.
- 현재 상품 조회 API는 `/api/Products?rankId=...&syncDate=...`로 구현되어 있다.
- 현재 상품 저장 API는 `/api/Products/saveWithSequentialId?keywordId=...&rankId=...&syncDate=...`로 구현되어 있다.
- 현재 상품 저장 메서드는 내부에서 공급자를 먼저 저장한 뒤 상품을 저장한다.
- 현재 `V3__create_supplier_and_product_table.sql` 기준으로 `product.sync_date DATETIME` 컬럼이 존재한다.
- `/api/Products/rpaSaveToday?size=20` 엔드포인트가 추가되어 당일 `TrendKeyword` 타깃을 한번에 처리할 수 있다.

## 3. 반복 처리 규칙
- 반복 시 조합할 값은 `keywordId`, `rankId`, `syncDate`다.
- `TrendKeyword -> Supplier -> Product` 순서로 처리한다.
- 하드코딩된 값은 예시로만 사용하고, 실제 구현에서는 동적으로 조회한다.
- 실제 구현에서는 `TrendKeyword.rankingId`를 `rankId`로, `TrendKeyword.syncDate`를 `syncDate`로 변환해 사용해야 한다.
- `SupplierAndProductService`는 `Constants.Snxbest.API_V1_FOOD_PRODUCT_DATA_URL_TEMPLATE`를 사용해 네이버 공급자/상품 API를 호출한다.
- `Product.syncDate`에는 해당 상품이 네이버 API의 어떤 동기화 날짜 기준으로 저장되었는지 기록해야 한다.
- RPA 반복 실행 시에는 `keyword`, `rank`, 당일 `crawled_at` 날짜 범위, 파싱된 `sync_date` 조합으로 이미 처리한 분류인지 먼저 판단한다.
- 이미 처리한 분류는 `SKIPPED`로 기록하고 공급자/상품 외부 API 호출 자체를 건너뛴다.
- 상품 개별 저장 중복 방지는 `keyword`, `rank`, 상품명, 당일 `crawled_at` 날짜 범위, 파싱된 `sync_date` 조합으로 판단한다.
- 같은 `syncDate`라도 다른 `keyword` 또는 `rank` 분류의 상품은 별도 분류 결과이므로 저장 대상이 될 수 있다.

## 4. 중단 재개 기준
- 중단 재개를 고려해 각 단계의 진행 상황을 기록한다.
- 당일 파싱한 `TrendKeyword` 목록을 기준으로 반복문을 돌린다.
- 현재 단계에서는 키워드별 실행 결과를 `COMPLETED`, `SKIPPED`, `FAILED`로 반환해 어느 키워드가 저장/스킵/실패되었는지 확인할 수 있다.

## 완료 기준
- 당일 `TrendKeyword` 목록을 기준으로 공급자와 상품 저장이 반복 실행된다.
- `size=20`이면 20개 상품을 강제로 채우는 것이 아니라 20개 키워드 분류를 각각 1회 처리한다.
- 샘플 파라미터가 아닌 실제 조회값으로 저장 API가 호출된다.
- 키워드별 결과는 `keywordId`, `rankId`, `syncDate`, `trendRank`, `status`, `productCount`, `message`로 확인할 수 있다.
- 판매자는 이름 기준으로 중복 저장되지 않고, 기존 판매자를 상품에 연결한다.
- 오늘 같은 키워드명, 같은 순위, 같은 `syncDate`가 이미 처리되어 있으면 대시보드 RPA 실행 버튼을 다시 눌러도 외부 공급자/상품 API를 재호출하지 않는다.
- 상품은 같은 키워드 분류에서 같은 상품명, 같은 `syncDate`, 같은 `crawled_at` 날짜에 이미 저장되어 있으면 다시 저장하지 않는다.

## 추가 구현 필요
- RPA 전체 실행 로그(`rpa_log`)와 파일 로그 저장은 아직 연결되지 않았다.
- 비동기 실행, 중복 실행 방지, 실패 재시도는 후속 단계에서 구현해야 한다.

## 이번 단계 구현 결과
- `RpaSupplierProductParseResult` record를 추가했다.
- `RpaSupplierProductParsingService.parseTodaySupplierAndProducts(int requestedSize)`를 추가했다.
- 기존 개별 API를 HTTP로 재호출하지 않고 `TrendKeywordService`, `TrendKeywordRepository`, `SupplierAndProductService`를 직접 사용한다.
- `SupplierAndProductService`의 날짜 가드를 조정해 같은 날짜의 여러 키워드 저장을 허용하고, 이전 날짜 데이터만 차단하도록 변경했다.
- 공급자 저장은 최신 날짜 비교 가드를 제거하고, 공급자 이름 기준 재사용 정책으로 정리했다.
- 상품 저장은 기존 `crawledAt` 전체 시각 일치 비교 대신 `keyword + rank + 상품명 + syncDate + crawledAt 당일 범위` 기준으로 중복을 검사한다.
- `SupplierAndProductService.hasTodayProductsForKeywordAndSyncDate`와 RPA 실행 전 스킵 가드를 추가해, 이미 처리된 키워드 분류는 외부 API 호출 전에 `SKIPPED` 처리한다.
