# 003 로그 저장

## 목적
- RPA 실행 상태와 상세 디버깅 정보를 DB 로그와 파일 로그로 분리해 남긴다.

## 1. DB 로그 저장
- DB에는 `message`를 통해 어떤 데이터가 저장되었는지 요약 기록한다.
- `rpa_log`의 `status`, `started_at`, `ended_at`, `keyword_count`, `product_count`를 함께 사용한다.
- 현재 `RpaLog` 엔티티와 `RpaLogRepository`는 존재한다.
- 현재 `RpaLog.builder(startedAt)`는 생성 시 상태를 `RUNNING`으로 설정한다.
- 현재 `RpaLog.complete(keywordCount, productCount)`와 `RpaLog.fail(message)`가 구현되어 있다.

## 2. 파일 로그 저장
- 파일 로그에는 RPA 실행 시점, 저장된 데이터, 디버깅 로그를 상세히 기록한다.
- 로그 파일 저장 위치는 `src/main/resources/static/test/rpa/log`다.
- 로그 파일명 규칙은 `rpa_parsing_yymmdd_time.log`다.
- 현재 `static/test/rpa/log` 폴더는 존재하지만, 파일 로그 작성 서비스는 확인되지 않았다.

## 3. 로그 역할 분리
- DB 로그는 운영 조회와 상태 추적용으로 사용한다.
- 파일 로그는 디버깅과 상세 추적용으로 사용한다.

## 완료 기준
- RPA 실행마다 `rpa_log`에 요약 상태가 남는다.
- 실행마다 지정된 위치에 상세 로그 파일이 남는다.

## 추가 구현 필요
- RPA 시작 시 `RpaLog`를 저장하고, 종료 시 같은 로그를 `COMPLETED` 또는 `FAILED`로 갱신하는 서비스가 필요하다.
- `RpaLog.id` 자동 생성이 주석 처리되어 있으므로 저장 전에 수동 ID 부여 방식이 필요하다.
- 현재 `RpaLogRepository`에는 최신 로그 조회만 있으므로, 날짜 범위 조회와 대시보드용 목록 조회 메서드가 필요하다.
- 파일 로그명에는 실제 날짜/시간 포맷 예시를 확정해야 한다. 예: `rpa_parsing_260506_1145.log`
