# 006 비동기 동시성 제어

## 목적
- 웹 요청과 실제 RPA 실행을 분리하고, 중복 실행으로 인한 충돌을 방지한다.

## 1. 비동기 처리
- `@Async`를 사용해 웹 요청과 실제 RPA 실행을 분리한다.
- `SemiApplication`에 `@EnableAsync`가 추가되었다.
- `RpaAsyncExecutionService.runSupplierProductParsingAsync(int requestedSize)`가 `@Async`로 실행된다.
- `/rpa/run?size=20`은 즉시 `202 Accepted`를 반환하고 실제 공급자/상품 파싱은 백그라운드에서 실행한다.

## 2. 동시성 제어
- 중복 실행 방지를 위해 낙관적 락(`@Version`)을 검토한다.
- 현재 `RpaLog`, `TrendKeyword`, `Supplier`, `Product` 엔티티에는 `@Version` 필드가 없다.
- 현재 단계에서는 `AtomicBoolean`과 최신 `RpaLog.status == RUNNING` 확인으로 중복 실행을 1차 차단한다.
- 실행 중 재요청이 들어오면 `/rpa/run`은 `409 Conflict`를 반환한다.

## 3. 종료 보장
- `try-finally` 구조를 사용해 성공/실패와 관계없이 종료 처리를 보장한다.
- 비동기 실행 서비스는 `finally`에서 메모리 실행 플래그를 해제한다.
- `RpaSupplierProductParsingService`는 큰 예외가 발생해도 `RpaLog`를 `FAILED`로 갱신하도록 보강되었다.
- `/rpa/status`는 최신 `RpaLog` 상태를 JSON으로 반환한다.

## 4. 재시도
- 외부 사이트 호출은 실패 가능성이 있으므로 재시도 전략을 둔다.

## 완료 기준
- RPA 실행 요청이 웹 요청과 분리되어 처리된다.
- 중복 실행과 예외 발생 시 상태 꼬임을 방지한다.
- `/rpa/run`으로 비동기 실행을 시작할 수 있다.
- `/rpa/status`로 최신 실행 상태를 조회할 수 있다.

## 추가 구현 필요
- 현재 중복 실행 방지는 단일 애플리케이션 인스턴스 기준의 `AtomicBoolean`과 DB 최신 상태 확인 조합이다.
- 다중 서버 배포까지 고려하려면 DB 락, 별도 실행 테이블, `@Version` 중 하나를 추가 검토해야 한다.
- 외부 사이트 호출 재시도는 아직 구현되지 않았다.

## 이번 단계 구현 결과
- `RpaAsyncExecutionService`를 추가했다.
- `RpaRunResponse`, `RpaStatusResponse`를 추가했다.
- `RpaRestController`의 `/rpa/run`, `/rpa/status` placeholder를 실제 응답으로 교체했다.
- `RpaSupplierProductParsingService`에 큰 예외 발생 시 `FAILED` 로그를 남기는 보강 처리를 추가했다.
